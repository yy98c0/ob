# 自动化发布与文件管理开发计划

## 目标

围绕当前 Quartz + Obsidian 的发布体系，逐步建立一条稳定、低摩擦、可回滚的自动化链路，重点覆盖以下两类能力：

1. **自动化发布**：从 Obsidian 写作、选择性发布、同步到 GitHub Pages，全流程尽量减少人工操作。
2. **文件管理**：保证 Vault 与 Quartz 仓库物理分离，避免草稿、任务书、私密文档误入发布目录，同时提升内容组织效率。

---

## 一、现状总结

当前仓库已经具备以下基础能力：

- Quartz v5 站点配置已存在
- GitHub Pages 自动部署 workflow 已配置
- TikZ 构建期渲染已接入
- 深色主题、背景动画、验收页已就位
- Obsidian Syncer 的使用说明已整理
- `content/` 下已有首页与测试页

因此后续开发不应再从零搭建，而应转入 **流程固化、边界收敛、可靠性增强**。

---

## 二、开发原则

### 1. 内容与工程分离

- Obsidian Vault 只负责写作与标记发布
- Quartz 仓库只负责构建、样式、插件和部署
- 不在两边之间做“全量复制”
- 不把需求文档、草稿、私密笔记混入发布内容

### 2. 单向发布

建议长期维持以下单向链路：

`Obsidian Vault -> 发布标记 -> Quartz content/ -> GitHub Actions -> GitHub Pages`

### 3. 默认拒绝误发布

- 默认不发布未标记笔记
- 默认忽略 `个人博客/`、`.obsidian/`、模板目录等
- 任何自动化脚本都必须带白名单或显式选择机制

### 4. 可验证优先

每个发布环节都要有可验证输出：

- 本地构建成功
- TikZ 输出为 SVG
- 公式渲染正常
- Pages 部署成功
- 最终线上页面可访问

## 三、后续开发计划

### 阶段 1：发布链路收口

目标是把“怎么发”变得清楚且安全。

#### 要做的事

- 确认并固化 `baseUrl`
- 明确 Obsidian Syncer 的发布目录和发布规则
- 将发布说明整理成一份短 SOP
- 补齐首页、测试页、发布页之间的链接关系
- 清理不必要的死链，比如未创建的测试页面引用

#### 交付物

- 一份最终版发布说明
- 一份简化的一键发布流程
- 一份可复用的验收页面结构

---

### 阶段 2：文件管理规范化

目标是让内容组织和发布范围可控。

#### 要做的事

- 为 Obsidian Vault 设计清晰的目录约定
  - 草稿区
  - 正式笔记区
  - 归档区
  - 项目/任务区
- 统一 frontmatter 规范
  - `title`
  - `description`
  - `tags`
  - `aliases`
  - 发布标记字段
- 建立“发布白名单”机制
- 对高风险目录做强制忽略
- 确保任务书、需求文档不参与发布

#### 建议规则

- 正式发布内容只放在指定知识目录中
- 任务书、规划文档保持独立
- 验收页单独维护，不混入正文知识流
- 图片、附件、代码示例按固定目录管理

#### 交付物

- 目录结构约定文档
- frontmatter 规范文档
- 发布白名单/黑名单规则说明

---

### 阶段 3：自动化发布增强

目标是减少人为操作，把发布变成“可重复动作”。

#### 当前已开始实现的内容

- 新增仓库内本地发布脚本 `scripts/publish-from-obsidian.mjs`
- 新增 `npm run publish:note` / `npm run publish:note:dry-run`
- 新增发布日志 `.quartz/publish-log.json`
- 强化对私密路径和不支持文件类型的拒绝机制

#### 下一步要做的事

- 设计 Obsidian 内的发布触发方式
  - 快捷键
  - 命令面板
  - 侧边栏按钮
- 将发布动作拆分成可观测步骤
  - 识别待发布文件
  - 检查内容合法性
  - 同步到 Quartz
  - 提交 Git 变更
  - 触发 Pages 部署
- 增加失败提示和回滚策略
- 对发布结果写入日志或状态文件

#### 重点改进

- 只发布当前笔记或已标记集合
- 发布前检查是否包含敏感路径
- 发布后检查线上构建状态
- 对失败步骤给出明确报错

#### 交付物

- 一套可执行的发布脚本方案
- 一套发布状态日志规范
- 一份失败恢复手册

---

### 阶段 4：构建与预览自动化

目标是减少“改完才发现坏了”的情况。

#### 要做的事

- 增加本地预检步骤
  - Markdown 结构检查
  - TikZ fence 检查
  - 公式块检查
  - 死链检查
- 支持本地预览前的最小构建
- 让测试页成为持续验收页面
- 若需要，可加入本地 watch / loop 机制

#### 交付物

- 本地构建检查清单
- 自动预览流程
- 验收页维护规范

---

### 阶段 5：可靠性与可维护性

目标是让系统长期可用，而不是短期能跑。

#### 要做的事

- 为 TikZ 缓存和编译失败增加更清晰的诊断信息
- 为发布脚本增加超时和重试
- 定期清理临时缓存与构建产物
- 记录各类常见失败模式
- 对 workflow 变更做版本化说明

#### 交付物

- 故障排查清单
- 缓存清理策略
- 发布异常处理规范

---

## 四、推荐实施顺序

建议按以下优先级推进：

1. **确认 `baseUrl` 与 Pages 地址**
2. **整理 Obsidian 发布目录和文件边界**
3. **补齐发布 SOP**
4. **完善发布白名单/黑名单**
5. **建立预检和验收页机制**
6. **再做发布自动化增强**
7. **最后优化日志、重试、缓存与维护策略**

---

## 五、文件管理建议结构

下面是一个推荐的 Obsidian 侧内容分层思路：

- `知识/`
  - 正式学术笔记
  - 可发布内容
- `草稿/`
  - 未完成文章
  - 不公开内容
- `项目/`
  - 工程性内容
  - 任务规划
- `归档/`
  - 已完成、较少修改的笔记
- `个人博客/`
  - 需求文档、发布规划、内部说明
  - 默认不发布

这套分层的核心是：**让“能不能发”从内容语义上就可判断，而不是完全依赖手工记忆。**

## 七、风险与注意事项

1. **误发布风险**
   - 这是当前最大的风险点
   - 必须靠目录边界和白名单机制解决

2. **构建失败风险**
   - TikZ、公式、死链都可能导致构建异常
   - 发布前应有本地预检

3. **配置漂移风险**
   - Obsidian 端和 Quartz 端规则可能不一致
   - 需要维护一份统一规范

4. **维护成本风险**
   - 自动化越多，日志与文档越重要
   - 否则后期排障会很困难

---

## 八、近期下一步建议

如果现在要立刻开始做，我建议按下面顺序推进：

1. 修正并确认 `baseUrl`
2. 检查 `content/index.md` 的链接是否存在死链
3. 明确 Obsidian 侧的发布目录规则
4. 补一份更短的“一键发布 SOP”
5. 再考虑做更强的脚本化自动发布

---

## 九、结论

当前系统已经具备静态博客自动化发布的基础设施。后续开发重点不应放在“再搭一个系统”，而应放在：

- **收紧文件边界**
- **明确发布范围**
- **固化操作流程**
- **增强失败可见性**
- **让自动化可长期维护**

这会比单纯增加功能更重要。
