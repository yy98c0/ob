# 自动化发布与文件管理技术总结

## 1. 背景

本项目基于 Quartz v5 构建个人学术博客，内容创作与站点发布采用 Obsidian + Quartz 的分离式工作流。为了降低手动同步成本，并确保内容发布的可控性与安全性，项目逐步建立了自动化发布与文件管理能力。

本技术总结记录当前已完成的基础层设计、实现方式、适用范围以及后续扩展方向，作为后续开发和维护的正式参考文档。

## 2. 目标

本阶段开发的目标是为自动化发布建立统一的基础设施，重点包括：

- 统一 Obsidian Vault 到 Quartz `content/` 的文件流转规则
- 限制私密内容误发布
- 为批量发布、发布前检查、发布日志和 CI/CD 接入提供可扩展接口
- 保持当前发布流程与 Quartz 构建系统兼容

## 3. 已完成的基础能力

### 3.1 统一文件策略模块

项目新增了 `scripts/file-policy.mjs`，用于集中管理文件发布相关的公共规则。该模块目前承担以下职责：

- 定义 Obsidian Vault 的默认路径
- 定义 Quartz `content/` 的目标目录
- 统一黑名单目录前缀
- 统一不允许发布的文件后缀
- 提供路径归一化函数
- 提供发布权限校验函数
- 提供目标文件路径生成函数

通过该模块，原本分散在多个脚本中的规则得以集中管理，降低了维护成本。

### 3.2 发布与拉取脚本重构

以下脚本已改为复用统一文件策略模块：

- `scripts/pull-from-obsidian.mjs`
- `scripts/publish-from-obsidian.mjs`

重构后，这些脚本共享相同的路径校验逻辑和目标路径生成逻辑，避免了规则不一致的问题。

### 3.3 已保留的发布能力

当前脚本仍支持以下常用能力：

- 单文件发布
- 多文件发布
- `--dry-run` 预演模式
- `--list` 批量清单模式

这意味着现有的手工发布习惯可以继续使用，同时也为进一步自动化保留了接口。

## 4. 技术实现概述

### 4.1 文件策略设计

文件策略模块采用“规则集中、脚本复用”的设计思路。主要规则包括：

- **允许发布的来源范围**：Obsidian Vault 内的内容
- **禁止发布的目录**：如 `个人博客/`、`.obsidian/`、`templates/`、`tmp/`、`drafts/`
- **禁止发布的文件类型**：如 `.canvas`、`.exe`、`.dll`、`.dylib`、`.so`

该策略的核心目的，是确保发布系统只处理适合公开发布的 Markdown 内容，避免私密笔记、配置文件和非文本资源误入站点仓库。

### 4.2 内容归一化

在将 Obsidian 内容写入 Quartz `content/` 时，脚本会进行基础归一化处理，包括：

- 将 ` ``` tikz ` 形式的代码块统一规范为 ` ```tikz `
- 对某些 `array` 形式的数学环境进行兼容性处理

这样可以提高 Quartz 构建阶段对公式与 TikZ 代码块的识别成功率。

### 4.3 安全控制

发布脚本在执行前会对路径和文件类型做显式检查，避免以下问题：

- 误发布私人目录
- 从 vault 外部写入内容
- 发布不适合进入站点仓库的二进制或特殊文件

这种控制方式虽然简单，但足以作为当前阶段的安全边界。

## 5. 与现有站点架构的关系

当前项目的整体发布链路可以概括为：

1. 在 Obsidian 中完成内容创作
2. 使用同步/发布脚本将选定内容写入 Quartz `content/`
3. 由 Quartz 构建站点
4. 通过 GitHub Actions 部署到 GitHub Pages

本次实现的文件管理基础层主要位于第 2 步，它确保进入站点仓库的内容是受控、可追踪且符合发布标准的。

## 6. 当前能力边界

当前阶段已经具备以下能力：

- 内容发布路径清晰
- 黑名单规则集中管理
- 发布脚本可复用
- 能进行 dry-run 验证
- 能对批量发布清单进行处理

但目前仍未完全实现以下增强功能：

- 自动生成正式发布清单
- 发布前完整校验报告
- 发布审计日志的结构化分析
- 更细粒度的目录白名单机制
- 与 Git 操作的全自动联动

## 7. 后续扩展建议

为了进一步完善自动化发布系统，建议按以下顺序继续开发：

### 7.1 发布前检查模块

增加独立的检查脚本，在写入 `content/` 前完成以下验证：

- 路径是否合法
- 文件是否存在
- 是否属于可发布范围
- 是否存在不支持的文件类型

### 7.2 发布清单机制

支持将待发布内容整理为独立清单文件，便于：

- 批量发布
- 审核发布范围
- 避免手动逐个输入路径

### 7.3 发布日志模块

记录每次发布的元信息，例如：

- 发布时间
- 来源路径
- 目标路径
- 发布模式
- 文件体积

### 7.4 Git 自动化联动

在内容写入与构建验证通过后，可进一步将流程联动到：

- `git add`
- `git commit`
- `git push`

以实现真正的半自动或全自动发布闭环。

## 8. 工程评价

从当前阶段来看，本项目已经完成了自动化发布体系的第一层基础设施建设：

- 文件策略统一
- 发布脚本复用
- 内容边界清晰
- 安全规则明确

这一层的价值不在于“做了多少功能”，而在于“为后续自动化留下了稳定、可维护的结构”。

## 9. 结论

本次开发将原本分散的 Obsidian 到 Quartz 内容流转逻辑，整理成了一个可扩展的文件管理基础层。它为后续实现批量发布、发布审核、日志审计以及 Git 自动提交提供了可靠起点。

如果后续继续推进，该体系可以逐步演进为一个完整的个人知识博客自动化发布管线。
